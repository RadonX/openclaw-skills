[assistant] **Selfware Protocol** — Agent 时代的统一文件协议草案。

核心设计哲学：

| 原则 | 含义 |
|------|------|
| **A file is an app** | 一切即文件，单文件或 `.self` 容器承载数据+逻辑+视图 |
| **去中心化协作** | 人↔Agent、Agent↔Agent 的协作不依赖中心化服务 |
| **Canonical Data Authority** | 每个实例必须定义单一数据来源（防止冲突） |
| **Write Scope Boundary** | 写入限制在 `content/` 或声明范围内（安全边界） |
| **No Silent Apply** | 更新必须经用户确认（不可静默覆盖） |
| **View as Function** | `View = f(Data, Intent, Rules)` — 视图是派生，非真相 |

---

**与 Trellis/Pensieve 的关系**：

| 工具 | 层级 | 关系 |
|------|------|------|
| **Trellis** | 工作流框架 | 可运行在 Selfware 容器内？ |
| **Pensieve** | 记忆系统 | 可成为 Selfware 的 "Memory" 模块实现？ |
| **Selfware** | 协议层 | 定义文件格式与协作规范，不限定上层工具 |

Selfware 试图定义 **Agent 时代的 "文件系统标准"**，而 Trellis/Pensieve 是特定场景的 **工作流实现**。如果 Selfware 成熟，Pensieve 的 `maxims/decisions` 可能以 `.self` 容器形式存在。

## 1. "A file is an app" → `.self` ZIP 容器

```bash
my_project.self          # ZIP 格式
├── self/manifest.md     # 必须：容器元数据
├── selfware.md          # 协议定义（可选本地覆盖）
├── content/             # 权威数据（Canonical Data）
│   ├── selfware_demo.md
│   └── memory/          # 可选：审计日志
├── views/               # 渲染视图（函数化）
│   ├── doc.md
│   ├── canvas.md
│   └── mindmap.md
└── server.py            # 本地运行时（Kernel）
```

**关键**：`.self` 是普通 ZIP，任何解压工具可打开；`self/manifest.md` 声明协议版本、数据来源、协作后端。

---

## 2. Write Scope Boundary → 运行时强制沙箱

| 区域 | 写入权限 | 说明 |
|------|---------|------|
| `content/` | ✅ 允许 | Canonical Data 区域，用户内容 |
| `selfware.md`<br>`manifest.md` | ❌ 默认禁止 | 协议定义文件，除非用户显式授权 |
| `kernel/`<br>`views/` | ❌ 禁止 | 运行时实现，可被 Agent 替换 |

**API 边界**：
```python
POST /api/save  # 只能写 content/ 下的文件
```

---

## 3. No Silent Apply → 更新确认流程

```python
if check_update():
    # 必须展示
    1. 更新来源 + 对比逻辑
    2. 变更摘要（diff/changelog）
    3. 创建回滚点（Git commit/tag）
    4. 用户确认：Accept / Reject / Defer
    
    apply_update()  # 仅在 Accept 后执行
```

## 5. Canonical Data Authority → 单源真相声明

```yaml
# manifest.md
Canonical-Data-Scope: content/      # 本地权威
# 或
Remote-Canonical: git:https://github.com/xxx
Remote-Policy: sync|merge          # 策略声明
```

**约束**：
- 运行时启动时必须确认数据来源
- 远程变更必须遵循 No Silent Apply
- 冲突时必须用户介入选择策略
