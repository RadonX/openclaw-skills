## Episode Summary

User was working on zsh dotfiles with stow and asked the agent to write an `INTENT.md` describing how to generate the zsh config. After several rounds of corrections — INTENT.md was too verbose, too procedural, included unnecessary metadata headers — the user pivoted: instead of fixing INTENT.md inline, create a reusable **skill** that distills any session into a minimal INTENT.md. The agent attempted skill creation via skill-distiller, which the user rejected as too specific, then re-ran with skill-distiller-gemini. Through iterative refinement (msgs 24–41), the skill's core contract was established: zero-arg, proposal-only, minimal output (Goal + invariants), no questions.

## Key Decisions / Constraints

- **Zero-arg interface**: no CLI flags, no path arguments — derive everything from current session history
- **Proposal-only**: print INTENT.md content, never write files
- **Output minimalism**: only `# <title>`, `Model:`, `Updated:`, `## Goal`, `## Target state (invariants)` — anything procedural must be rewritten as invariant or deleted
- **No questions by default**: if session content is insufficient, emit a short error instead of asking
- **Header rejection**: user explicitly removed Audience, Author LLM, Model metadata lines from INTENT.md output — "these are not needed"
- **Description vs guardrails split**: frontmatter description kept routing-only; behavioral constraints live in SKILL.md guardrails
- **skill-distiller critique**: user questioned whether skill-distiller/skill-distiller-gemini were adequate for this workflow ("为啥 intent-writer skill create 起来这么困难")
