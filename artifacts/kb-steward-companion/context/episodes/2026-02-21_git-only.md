## Episode Summary

The kb-steward-companion skill was created from scratch as a set of CLI scripts to query and maintain an Obsidian vault managed by `kb-steward`. Over six commits across two weeks, it evolved from a bare script collection into a documented toolset with proper separation of concerns. The skill was eventually renamed to `kb-steward-tools` for clarity, though this episode covers the entire arc under the original name.

## Key Decisions / Constraints

- **No hardcoded vault paths**: Scripts auto-discover vault location from `obsidian-cli` configuration, ensuring portability across machines
- **Clear scope boundary**: kb-steward writes/ingests content; companion tools only query and maintain — never ingest
- **Safe frontmatter operations**: Use `obsidian-cli` (and later `yq` for tag.sh) rather than raw `sed`/`grep` for YAML manipulation
- **Rename to kb-steward-tools**: Renamed from "companion" to "tools" for simplicity; docs simplified from 154 to 61 lines
- **Dynamic tag discovery**: Removed static TAGS.md; scripts/tags.sh discovers tags at runtime instead
- **Delete with backup**: delete.sh creates `.bak` copies before deletion as a safety measure

## Relation to skill evolution

This is a clear `evolution` episode — the skill was built incrementally with each commit making structural changes: adding scripts, reorganizing documentation, clarifying the relationship with the parent `kb-steward` skill, and ultimately renaming the folder. Multiple evolution signals are present: new companion scripts were proposed and added, documentation structure was changed, and the skill's interface (what scripts exist and how they work) was actively shaped across sessions.
